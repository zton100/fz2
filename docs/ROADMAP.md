# 装备挂机游戏后续开发路线

## 项目判断

当前版本已经完成装备、挂机、锻造、离线和转生的 Demo 闭环，也具备可辨识的暗黑战斗界面。下一阶段不再继续单独换皮，而是让关卡内容、战斗事件、装备差异和联网工程共同支撑长期体验。

## 开发原则

- 按玩家可体验的纵向切片交付，每个里程碑同时包含领域逻辑、协议、客户端表现和测试。
- 服务端保持战斗结果和关键奖励权威，客户端负责动画编排和即时反馈。
- 新增装备和词缀前，先保证每项属性真实生效、评分与服务端结果一致。
- 外部测试前完成身份、存档、连接和数据容量硬化，不以本地 Demo 标准直接上线。
- 每个里程碑完成后运行 Go 单测、静态检查、协议验证和 Unity PlayMode/视觉验证。

## M1：完整关卡与战斗事件

目标：把“一次战力比较直接推层”改为“小兵波次 -> 守关敌人 -> 推进下一层”。

- 每层包含固定数量的小兵，击杀进度由服务端保存。
- 普通层最后出现精英守卫，每 5 层出现 Boss。
- 小兵不制造整包装备洪水；守关敌人提供主要装备和首通奖励。
- 服务端推送攻击结算、击杀、波次进度、守关入场和通关事件。
- 客户端实现攻击前摇、突进、受击闪烁、伤害飘字、死亡和掉落反馈。
- 在线、离线和重连使用同一套关卡推进规则。

验收：新角色可以完整看到小兵进度、守关入场、通关掉落和下一层切换；第 5 层 Boss 有独立反馈；重连后进度不倒退。

## M2：可信的装备决策

目标：解决客户端评分与服务端真实战力不一致、部分词缀无实际效果的问题。

- 服务端提供装备替换后的真实战力差值或统一评分。
- 一键穿戴、提升提示、弱装分解全部使用权威结果。
- 为敏捷、智力、体力、吸血、命中、闪避、护盾等属性补充战斗作用。
- 暂时无法进入玩法闭环的词缀从掉落池移除，不保留纯展示属性。
- 增加构筑标签和关键属性摘要，降低装备比较成本。

验收：客户端所有“提升/更弱”判断与实际穿戴后的战力方向一致；掉落池中不存在无效果词缀。

## M3：装备、怪物与关卡内容扩充

目标：让中短期掉落和推层持续出现可识别的新内容。

- 装备基底由每槽位 1 件扩展为首批 40 件以上，并外置为版本化数据。
- 每个基底拥有独立图标和属性倾向，边框、强化和锁定采用可复用 UI 叠层。
- 首批实现 12 件传奇装备、4 件神器及其特殊机制。
- 增加普通怪、精英和 Boss 家族，配置不同攻防倾向与奖励池。
- 增加关卡环境和阶段词缀，避免无限层只变化数值。

验收：连续 30 层至少出现多个怪物家族、装备系列和可改变构筑选择的传奇效果。

## M4：长期目标与联网价值

目标：从联网存档 Demo 发展为具有回归和比较动力的联网挂机游戏。

- 增加引导、任务、成就和装备图鉴。
- 扩展转生为分支天赋和里程碑解锁，不再只有四项线性数值。
- 增加玩家档案和排行榜，随后接入周期 Boss 或赛季目标。
- 建立活动配置、奖励邮件和基础管理能力。

验收：玩家在单次上线、每日回归和长期赛季三个时间尺度上都有明确目标。

## M5：外测工程硬化

目标：消除账号、存档、网络和容量方面的上线阻断风险。

- 接入真实身份认证；校验账号标识并消除存档路径穿越风险。
- 限制 WebSocket Origin、消息大小和请求频率，增加心跳、超时和统一错误响应。
- 客户端增加重连、退避、请求状态、断线重同步和前后台恢复。
- 背包增加容量、批量整理、增量协议和虚拟化列表；离线收益改为批量摘要。
- 持久化增加 schema 版本、迁移、损坏隔离、备份和优雅停机，逐步迁移数据库。

验收：断网、重复登录、异常消息、8 小时离线、大背包和服务重启均有自动化验证。

## M6：发布与运营基线

- 建立 CI：格式化、Go test/vet/race、协议测试、Unity 编译与 PlayMode smoke。
- 建立环境配置、容器化部署、健康检查、日志、指标、告警和备份恢复演练。
- 完成目标平台构建、版本号、资源清单、隐私条款和账号删除流程。
- 执行负载、长稳、兼容性和首轮小规模外测。

验收：测试、构建和部署可重复执行，关键线上状态可观测并可恢复。

## 当前执行顺序

1. M1 服务端关卡状态与行为测试。
2. M1 战斗事件协议和客户端动效。
3. M2 权威装备评分与有效词缀。
4. M3 装备数据和图标生产管线。
5. M5 外测硬化。
6. M4 长期内容与 M6 发布能力按外测反馈推进。
